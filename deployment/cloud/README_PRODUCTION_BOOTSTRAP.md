# Production Bootstrap Runbook

This runbook initializes the first real cloud organization, store, and owner
account after a fresh cloud database has been created.

Use this only once per production database.

## What This Does

`bootstrap-admin.sh` creates, in one database transaction:

- `organizations`
- `stores`
- first owner `users` row
- `user_credentials` with a BCrypt password hash generated by backend
  `PasswordService`
- `organization_memberships`
- `store_memberships`
- missing `OWNER` role metadata only if the role does not already exist

It does not create menu data, demo data, dining tables, printer configs, or
default development accounts.

## What This Does Not Enable

Do not turn on these flags for production bootstrap:

- `app.seed.default-users-enabled`
- `app.seed.demo-data-enabled`
- `app.seed.membership-supplement-enabled`
- `app.seed.production-bootstrap-enabled`

`bootstrap-admin.sh` explicitly runs with those flags disabled.

## Safety Rules

- The script refuses to run if any active production owner/admin already exists.
- It never overwrites users, credentials, memberships, or passwords.
- Password input is hidden and is passed to a temporary backend process through
  stdin only.
- The password is not written to shell history, `.env`, logs, command-line
  arguments, or temporary files.
- `bootstrap-admin.env` is optional and must contain only non-password fields.
  The script parses this file itself and does not pass it to Docker Compose.
- The backend hashes the password with the existing BCrypt password service.
- The temporary bootstrap container is removed with `docker compose run --rm`.
- The normal cloud backend container continues running.

## Before Running

Make sure deployment is healthy:

```bash
cd /opt/restaurant-system/deployment/cloud
./health-check.sh
```

Back up the empty or near-empty database anyway:

```bash
./backup-db.sh
```

## Optional Env File

To avoid typing organization/store/owner fields repeatedly, create a local
bootstrap file on the server:

```bash
cd /opt/restaurant-system/deployment/cloud
nano bootstrap-admin.env
chmod 600 bootstrap-admin.env
```

Example:

```text
BOOTSTRAP_ORGANIZATION_NAME=Lanzhou Noodles
BOOTSTRAP_STORE_NAME=4483 R. Saint-Denis
BOOTSTRAP_OWNER_USERNAME=owner
BOOTSTRAP_OWNER_FULL_NAME=Chuwen Huang
BOOTSTRAP_OWNER_EMAIL=
BOOTSTRAP_OWNER_PHONE=
```

Do not add `BOOTSTRAP_OWNER_PASSWORD`. The script rejects that field because the
password must not be saved in this file.

## Dry Run

Use dry-run first. It validates inputs and confirms the one-time owner/admin
guard without writing data:

```bash
./bootstrap-admin.sh --dry-run
```

With a local bootstrap file:

```bash
./bootstrap-admin.sh --dry-run --env-file bootstrap-admin.env
```

You will be prompted for:

- organization name
- store name
- owner username
- owner full name
- owner email/phone, stored in `users.phone` because the current user model has
  no email column
- owner password
- password confirmation

Expected success output:

```text
organization: ...
store: ...
username: ...
bootstrap dry-run success
```

## Real Bootstrap

Run:

```bash
./bootstrap-admin.sh
```

With a local bootstrap file:

```bash
./bootstrap-admin.sh --env-file bootstrap-admin.env
```

Expected success output:

```text
organization: ...
store: ...
username: ...
bootstrap success
```

The script must not print the plaintext password or password hash.

## Password Stdin Mode

For automation where the password comes from a password manager or another
secure local command, use `--password-stdin` with `--env-file`:

```bash
your-password-manager-command \
  | ./bootstrap-admin.sh --env-file bootstrap-admin.env --password-stdin
```

This mode reads the password once from stdin and does not ask for confirmation.
Do not put the password in command history. Prefer a password manager command or
an interactive hidden prompt when possible.

## First Login

Open the site:

```text
http://YOUR_SERVER_PUBLIC_IP
```

or, if HTTPS is enabled:

```text
https://YOUR_DOMAIN
```

Log in with:

- username: the owner username entered during bootstrap
- password: the owner password entered during bootstrap

After login, confirm the owner workspace and store workspace load.

## After Bootstrap

Configure production data intentionally:

- menu categories/items/options through Owner Admin or reviewed import
- dining tables through Owner Admin or reviewed import
- printing mode through Print Center

Printing guidance:

- Keep `DISABLED` if the restaurant is not ready to print.
- Use `MOCK` for cloud dry-run validation.
- Use `PAD_DIRECT` or a local print bridge for real store printers.
- The cloud backend must not use `REAL` to directly connect to private LAN
  printers.

## If Bootstrap Fails

- Read the error.
- Do not enable demo seed flags.
- Do not manually edit password hashes.
- Fix the input or database condition and rerun dry-run first.
- If partial data somehow appears, restore the pre-bootstrap backup before
  trying again.

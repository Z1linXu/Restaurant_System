# 🍜 Restaurant Management System - MVP Scope

---

# 1. 背景（Background）

本项目旨在从 0 到 1 构建一套餐厅管理系统，用于提升餐厅日常运营效率，减少人工错误，并为管理层提供数据支持。

当前市场中存在的问题：
- 点单与后厨沟通不顺畅（错单/漏单）
- 前台拖单严重（一次积攒四五张单）
- 高峰期后厨混乱，缺乏清晰分工
- 上菜时前台不清楚哪个菜是哪个桌子，容易出错
- 库存管理依赖人工经验，容易出错
- 缺乏统一的数据分析工具

本系统目标是打造一个**前台 + 后厨 + 库存 + 数据分析的一体化系统**。

---

# 2. 产品目标（Goals）

MVP 阶段只关注核心闭环：

## 核心目标
- 提升点单效率与准确率
- 提高后厨出餐效率
- 实现订单 → 后厨 → 出餐 的完整流程闭环
- 实现基础库存自动扣减
- 提供基础经营数据（营收 & 菜品）

### 5️⃣ 出餐管理（Fulfillment / Ready Station）

#### 场景说明
后厨完成制作后，抓码人员将餐品放到出餐台，并在系统中将对应订单或菜品标记为“已完成 / Ready”。

出餐台会配备一个专门的显示器（Ready Screen），供服务员或前台查看当前已完成、等待取走的餐品。

服务员根据显示器内容前往出餐台取餐，并完成送桌或交付给顾客的动作。取走后，服务员需要在系统中将该订单标记为“已取走”或“已完成”。

---

#### 功能
- Ready Screen 只显示当前已完成、待取走的订单或菜品
- 抓码人员可将订单/菜品标记为 Ready
- 服务员可查看待取走列表
- 服务员取餐后可标记为已取走 / 已完成
- 支持按照订单号、桌号、自取号进行识别
- 支持按时间排序，优先显示最新完成或等待时间较长的订单

---

#### 建议状态流转

```text
In Preparation → Ready on Pickup Station → Picked Up / Served → Completed 
```

### 6️⃣ 门店库存与备菜管理（Inventory & Prep Management）

#### 场景说明
库存管理不仅用于后台记录原料数量，还需要支持门店现场的备菜决策。

抓码或备菜人员在空闲时，需要查看关键原料或半成品的当前剩余量、今日消耗量、昨日销售参考等信息，以判断是否需要继续备菜。

系统需要支持不同类型的库存项：
- 可精确计数型（如卤鸡蛋）
- 可精确计量型（如汤底、酱汁，单位为 ml / g）
- 估算型（如毛豆、小菜等，无法严格逐个计数）

---

#### 功能
- 原料 / 半成品管理
- BOM（菜品配方）管理
- 基于订单自动扣减库存
- 安全库存设置
- 低库存提醒
- 今日消耗量统计
- 昨日销售 / 消耗参考
- 备菜建议显示
- 支持抓码 / 备菜人员查看关键库存项

---

#### 库存项类型

##### 1. 计数型
适用于按“个 / 份”进行管理的库存项，例如卤鸡蛋。
每卖出一个相关菜品，系统自动扣减对应数量。

##### 2. 计量型
适用于按“ml / g / kg”进行管理的库存项，例如汤底、酱汁。
系统根据菜品规格和 BOM 自动扣减相应用量。

##### 3. 估算型
适用于难以严格精确计量的库存项，例如毛豆、小菜。
系统支持按经验份数或批次进行估算管理。

---

#### 备菜辅助示例

- 卤鸡蛋：
  - 今日已售：38 个
  - 当前剩余：12 个
  - 安全库存：15 个
  - 建议：补煮 30 个

- 素面汤底：
  - 今日已消耗：3200 ml
  - 当前剩余：900 ml
  - 安全库存：1000 ml
  - 建议：补做 3000 ml

---

#### MVP 建议
MVP 阶段先支持：
- 自动扣库存
- 安全库存提醒
- 今日消耗量展示
- 昨日参考销量展示
- 简单备菜建议

高级预测功能（如基于时段销量自动预测未来缺货）可放到后续版本。

# 6. 核心业务流程（Core Flows）

---

## 6.1 点单 → 后厨 → 出餐
前台创建订单 → 提交订单 → 系统拆单 → 后厨制作 → 标记完成 → 出餐 → 完成订单

---

## 6.2 库存流程
订单完成 → 按 BOM 扣减库存 → 判断库存 → 触发低库存提醒
